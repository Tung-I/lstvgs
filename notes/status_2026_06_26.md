# Project Status — 2026-06-26

**Goal:** Build a **standards-native adaptive streaming system** for stadium/city-scale 3DGS scenes — analogous to how MPEG-DASH handles video: serve only what the client needs (spatial locality) at the quality the bandwidth allows (LOD), using standard HTTP/WebSocket transport rather than proprietary protocols.

| Method | Contribution to Streaming |
|--------|--------------------------|
| Vanilla 3DGS | Baseline quality and scale reference |
| CityGS | **Spatial decomposition at training time** — block-parallel training so city-scale scenes converge |
| L3GS | **Importance-based LOD** — layer each view by contribution; stream coarser layers first |
| LapisGS | **Resolution-pyramid LOD** — train each layer at a different camera resolution; compact incremental representation |
| EvoGS | **Evolution-tree LOD** — replace parents with compact child pairs (ψ, α residuals); smaller total storage than LapisGS cumulative |
| SGSS | **ILP-optimized spatial cuboid streaming** — post-training partition into cuboids that minimize transmission waste for viewport-adaptive streaming |

**Interpretation issue:** The "LOD ladder" is only monotonically improving when each layer is evaluated at its *own* training factor. At a fixed full resolution, L0 (trained at 8× downsample) scores only 17 dB — far below paper expectations. The paper (LapisGS) reports L3 SSIM 0.913 for Garden, but our L3 at full res shows 0.744. This gap is **unexplained** — it may be:
1. Our anchor-freezing implementation differs from the paper's exact mechanism
2. Number of training steps or learning rate scheduling differs
3. The paper evaluates differently (only the newest enhancement layer's Gaussians?)

**Recommended next step:** Compare our `train_lapisgs.py` anchor-freeze approach to the official LapisGS repo implementation to identify any discrepancy. The training is done; re-running eval is fast.

### EvoGS — Garden ✅ DONE (2026-06-26)

**Code:** `lctvgs/evogs/`
```
evogs/
  __init__.py
  evolution_tree.py     # EvolutionTree class (splits, residuals, serialization)
  train_evogs.py        # main trainer (entry point)
  eval_evogs.py         # standalone evaluator
  scripts/train_garden_evogs.sh  # run script (fixed lctvgs-copy → lctvgs)
```
**Results dir:** `results/garden_evogs/`

**What EvoGS is:** Binary evolution tree — parents are **replaced** by two symmetric children (C1=P+ψ, C2=P−ψ). Contrast with LapisGS which adds layers without removing parents. 4 levels, 30K steps each, factors 8→4→2→1. Evolution splits every 3K steps until step 15K (5 events/level), top 1% of active leaves by 2D gradient norm.

**Results (each level evaluated at its native training factor — confirmed by paper):**

| Level | Factor | #GS | PSNR | SSIM | LPIPS | Leaves MB | Tree MB |
|-------|--------|-----|------|------|-------|-----------|---------|
| L0 | 8× | 2.84M | 29.63 | 0.922 | 0.032 | 669.7 | 0.0 |
| L1 | 4× | 2.63M | 27.83 | 0.868 | 0.075 | 620.0 | 35.0 |
| L2 | 2× | 2.61M | 26.74 | 0.813 | 0.152 | 615.4 | 34.9 |
| L3 | 1× | 2.61M | **26.43** | **0.795** | **0.238** | 615.5 | 34.9 |

**vs. LapisGS (our impl, same native-factor evaluation):**

| Level | Factor | LapisGS PSNR | EvoGS PSNR | Δ PSNR |
|-------|--------|-------------|-----------|--------|
| L0 | 8× | 29.64 | 29.63 | −0.01 |
| L1 | 4× | 26.99 | 27.83 | **+0.84** |
| L2 | 2× | 25.96 | 26.74 | **+0.78** |
| L3 | 1× | 25.60 | 26.43 | **+0.83** |

**Average improvement L1–L3: +0.82 dB** (paper claims +0.31 dB).

**Why the delta is larger than paper:** Our LapisGS baseline is weaker than the paper's (L3 SSIM=0.744 vs paper's 0.913). EvoGS and LapisGS are both below paper's absolute numbers, but EvoGS degrades less. The evaluation protocol is confirmed correct (native factors per level).

**Storage note:** Our L3 leaves are 615.5 MB (uncompressed float32, 2.61M Gaussians × ~236 bytes). Paper claims 347 MB. Discrepancy likely due to: (a) paper's scene yields fewer Gaussians, or (b) paper reports compressed sizes. The relative storage advantage vs LapisGS cumulative (1195 MB → 650 MB, −45%) is significant and directionally consistent with the paper.

**PLY files:** `results/garden_evogs/levels/level_{00..03}_leaves.ply`  
**Trees:** `results/garden_evogs/levels/level_{01..03}_tree/{topology.json,residuals.npz}`  
**Eval:** `results/garden_evogs/eval/metrics.json`

**Bug already fixed (do not re-introduce):** numpy scalar in residual extraction — parent snapshots stored with `np.atleast_1d(...).astype(np.float32)` in `evo_split()`, residuals use `.flatten()` before arithmetic.

### SGSS — Rubble 🔶 Stages 1–3a Done; Resume from Stage 3b

**What SGSS is:** A **post-training** streaming optimizer (ACM MM 2025). Takes a pre-trained PLY and uses Integer Linear Programming to find the optimal spatial cuboid partition that minimizes expected transmission waste for viewport-adaptive streaming. Does NOT modify training.

**Code:** `lctvgs/sgss/` AND `~/lctvgs-copy/sgss/` — both in sync.
```
sgss/
  run_sgss.py           # end-to-end orchestration (resumable)
  voxel_gaussian.py     # Stage 1: uniform 10×3×10 voxelization
  build_matrix_a.py     # Stage 2: 18,150 cuboid combinations
  cameras_from_colmap.py # Stage 3a: COLMAP → camera JSON
  projection_model.py   # Stage 3b: frustum visibility → Cd_new.npy
  run_ilp.py            # Stage 4: scipy.optimize.milp ILP
  optimal_voxelization.py # Stage 5: extract selected cuboids as PLYs
  streaming_cuboids.py  # Stage 6: PLY → .splat binary (SH 0–3)
  scenes.py             # shared scene config registry
```

## Known Issues / Pitfalls

- **Empty points3D:** rubble/sparse/0 and building/sparse/0 have 0 SFM points → always use `cityGSdata/colmap_results/` as SFM source
- **Opacity reset during block finetuning:** `--strategy.reset-every 3000` (default) destroys coarse quality; always use `--strategy.reset-every 100000` for blocks
- **COLMAP triangulation:** installed binary doesn't support `--Mapper.min_num_inliers`; don't use it
- **rubble_citygs images_4_png:** image 000215.png was 0 bytes (fixed manually with PIL resize to 1152×864)
- **LapisGS per-factor vs fixed-factor eval:** `metrics.json` uses per-layer training factor; `metrics_fixed_factor1.json` is all-layers at full res; the latter is the fair comparison for streaming quality
- **EvoGS numpy scalar bug:** fixed — parent snapshots stored with `np.atleast_1d`, residuals use `.flatten()` before arithmetic
- **SGSS parse_images_txt:** blank 2D-point lines must NOT be filtered before processing pairs (fixed)
